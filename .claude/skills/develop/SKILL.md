# TDD 개발 워크플로우

설계 문서(To-do + API/DB 설계)를 입력받아 구현을 진행한다.
**설계 문서가 없다면 먼저 `/plan`으로 설계 문서를 작성해야 한다.**

각 단계는 Agent에게 위임하여 별도 컨텍스트에서 실행한다.
Agent 완료 후 결과를 확인하고 사용자 승인을 받은 뒤 다음 단계로 진행한다.
**절대로 단계를 건너뛰지 않는다.**

## 필수 읽기 문서

| 상황 | 문서 |
|-----|-----|
| 항상 | `docs/DEV_COMMANDS.md` - 테스트/빌드 명령어 |

---

## Step 0: 설계 문서 파악

인자로 전달받은 설계 문서(`.claude/skills/plan/templates/design.md` 형식)를 읽고 다음을 파악한다:
- 배경 (왜 이 기능이 필요한가)
- To-do 리스트 (구현할 작업 목록)
- API 설계서 (엔드포인트, 요청/응답, 에러 케이스)
- DB 설계서 (테이블/컬럼 변경, 필요시)
- 테스트 관점 (테스트 시나리오)

**인자가 없는 경우:** 사용자에게 설계 문서의 경로 또는 링크를 질문한다.

> ✅ 설계 문서를 파악한 뒤 Step 1로 진행한다.

---

## Step 1: Test (TDD Red)
`test` Agent를 호출하여 테스트 작성을 위임한다.

**Agent에게 전달할 정보:**
- 설계 문서의 To-do 리스트
- API 설계서
- 테스트 관점 (시나리오)

**산출물:** `*Test.java` 파일

**검증:** `./gradlew test` 실행 → 테스트가 **실패**하는지 확인

> ✅ Agent 결과를 사용자에게 보여주고 확인을 받은 뒤 Step 2로 진행한다.

---

## Step 2: Implement (TDD Green)
`implement` Agent를 호출하여 구현을 위임한다.

**Agent에게 전달할 정보:**
- 설계 문서의 To-do 리스트
- API 설계서
- Step 1에서 작성된 테스트 파일 경로

**산출물:** 구현 코드

**검증:** `./gradlew test` 실행 → 모든 테스트 **통과** 확인

> ✅ Agent 결과를 사용자에게 보여주고 확인을 받은 뒤 Step 3으로 진행한다.

---

## Step 3: Review (코드 리뷰)
`review` Agent를 호출하여 코드 리뷰를 위임한다.

**Agent에게 전달할 정보:**
- 변경된 코드 파일 목록
- To-do 리스트 (변경 범위 확인용)

**산출물:** 리뷰 피드백 (Critical / Suggestion / Good)

**결과 처리:**
- Approve → 개발 사이클 완료! 노션 정리 + 커밋으로 마무리.
- Request Changes → Step 2로 돌아가 수정 후 재리뷰

> ✅ 개발 완료. 사용자에게 최종 결과를 보고한다.

---

## 마무리: 노션 정리 + 커밋

Review 완료 후:
1. 노션 섹션 상세 페이지 생성 (Part 1: 개념, Part 2: Q&A, Part 3: 면접)
2. 커리큘럼 페이지 완료 표시 (✅)
3. 커밋 (섹션 완료 시 규칙에 따라)
