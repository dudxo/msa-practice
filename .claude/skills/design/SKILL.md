# Design (API/DB 설계)

플래너의 To-do 리스트를 기반으로 API/DB 설계서를 작성한다.

## 입력

- 플래너가 작성한 **To-do 리스트** (컨펌 완료된 상태)
- 기존 API 스펙 (있다면)
- 기존 DB 스키마 (있다면)

## 출력

- **API 설계서** (사용자 컨펌 필수)
  - 엔드포인트 (경로, HTTP 메서드)
  - 요청/응답 스키마
  - 에러 케이스
- **DB 설계서** (필요한 경우)
  - 테이블/컬럼 정의
  - 관계 (1:N, M:N 등)
  - 인덱스 (필요시)
- **DTO 구조** (필요한 경우)

## 필수 읽기 문서

| 상황 | 문서 |
|-----|-----|
| 항상 | `docs/CONVENTIONS.md` - API 응답 포맷, 에러 코드 규칙 |
| 항상 | `docs/DOMAIN.md` - 서비스 개요 및 도메인 파악 |
| 기존 API 확인 시 | Controller 클래스 (`**/*Controller.java`) |
| 기존 DB 확인 시 | JPA Entity 클래스 (`**/entity/*.java`) |

## 행동 순서 (필수)

```
1. To-do 리스트 수신 (플래너에서)
       ↓
2. 기존 API/DB 구조 파악
       ↓
3. API 설계서 작성
       ↓
4. DB 설계서 작성 (필요시)
       ↓
5. 사용자 컨펌
```

> **설계 컨펌 없이 테스트/구현 단계 진행 금지**

### PASS 케이스

설계 변경이 필요 없는 경우 (버그 수정, 단순 리팩토링 등):

```markdown
## Design 판단

설계 변경이 필요하지 않습니다.

**이유**: 기존 API/DB 구조 변경 없이 로직 수정만으로 해결 가능

다음 단계(/test)로 진행해주세요!
```

## API 설계서 템플릿

```markdown
## API 설계서

### 1. [기능명]

#### `POST /api/v1/orders/:orderId/cancel`

**설명**: 주문을 취소합니다.

**요청**
| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|-----|------|
| orderId | string (path) | O | 주문 ID |
| reason | string (body) | X | 취소 사유 |

**응답 (200 OK)**
```json
{
  "orderId": "order_123",
  "status": "CANCELLED",
  "cancelledAt": "2024-01-15T10:30:00Z"
}
```

**에러 케이스**
| 코드 | 상황 |
|-----|-----|
| 400 | 잘못된 요청 (유효하지 않은 orderId) |
| 404 | 주문을 찾을 수 없음 |
| 409 | 이미 취소된 주문 |
```

## 행동 원칙

### DO
- `docs/CONVENTIONS.md`의 API 응답 포맷 준수
- RESTful 원칙 준수
- 기존 네이밍 컨벤션 따르기
- 영향 받는 기존 API가 있다면 명시
- 에러 케이스 빠짐없이 정의

### DON'T
- 사용자 컨펌 없이 다음 단계 진행
- 기존 API 스펙과 충돌하는 설계
- 불필요한 복잡도 추가
- 직접 코드 작성 (설계만!)

## 체크리스트

- [ ] API 엔드포인트가 RESTful한가?
- [ ] 요청/응답 스키마가 명확한가?
- [ ] 에러 케이스가 모두 정의되었나?
- [ ] `docs/CONVENTIONS.md` 규칙을 따르는가?
- [ ] 기존 API와 일관성이 있는가?
- [ ] DB 변경이 필요하면 설계했는가?
