# Plan (기획 분석 → To-do → 설계)

요구사항을 분석하고 개발 가능한 To-do 리스트와 API/DB 설계서를 작성한다.

## 입력

- 구두 설명 또는 기획서
- Slack/Notion 링크
- 버그 리포트

## 출력

**설계 문서** — 반드시 [templates/design.md](templates/design.md) 형식을 따른다.

**저장 위치** — 결과물은 `.claude/plans/` 디렉터리 안에 md 파일로 저장한다.
- 파일명 형식: `{기능명}-design.md` (예: `order-cancel-design.md`)

설계 문서에 포함되는 내용:
- 메타 정보 (기능명, 작성일, 도메인, 관련 이슈)
- 배경 (왜 이 기능이 필요한가)
- To-do 리스트 (사용자 컨펌 필수)
- API 설계서 (사용자 컨펌 필수)
- DB 설계서 (필요한 경우)
- 테스트 관점 정리

## 필수 읽기 문서
| 상황 | 문서 |
|-----|-----|
| 항상 | [templates/design.md](templates/design.md) - 설계 문서 템플릿 |
| 항상 | `docs/DOMAIN.md` - 서비스 개요 및 도메인 파악 |
| 항상 | `docs/CONVENTIONS.md` - API 응답 포맷, 에러 코드 규칙 |
| 기존 기능 수정 시 | `docs/features/{도메인}/feature.md` - 현재 동작 파악 |
| 기존 변경 이력 참고 시 | `docs/features/{도메인}/changes/change-{date}-{name}.md` |
| 기존 API 확인 시 | Controller 클래스 (`**/*Controller.java`) |
| 기존 DB 확인 시 | JPA Entity 클래스 (`**/entity/*.java`) |

## 행동 순서 (필수)

```
1. 요구사항 수신
       ↓
2. 질문 (필수!)
   - 모호한 부분 질문
   - 사용자 답변 대기
       ↓
3. To-do 리스트 작성
       ↓
4. 사용자 컨펌
       ↓
5. API/DB 설계 (To-do 기반)
   - 기존 API/DB 구조 파악
   - API 설계서 작성
   - DB 설계서 작성 (필요시)
       ↓
6. 사용자 컨펌
       ↓
7. 설계 문서 작성 (templates/design.md 형식)
   → .claude/plans/{기능명}-design.md 에 저장
```

> **Step 2를 건너뛰지 마세요!** 질문 없이 바로 To-do 작성 금지

### 설계 PASS 케이스

설계 변경이 필요 없는 경우 (버그 수정, 단순 리팩토링 등):

```markdown
## Design 판단

설계 변경이 필요하지 않습니다.

**이유**: 기존 API/DB 구조 변경 없이 로직 수정만으로 해결 가능
```

## 질문 가이드

### 필수 질문 카테고리

| 카테고리 | 질문 예시 |
|---------|----------|
| **범위** | "이 기능은 어떤 서비스에 영향을 주나요?" |
| **조건** | "어떤 조건에서 동작해야 하나요? (상태, 권한 등)" |
| **예외** | "예외 상황은 어떻게 처리하나요? (실패 시, 장애 시 등)" |
| **연동** | "다른 서비스와 통신이 필요한가요? (동기/비동기)" |

### 질문 작성 원칙

- 한 번에 3~5개 질문 (너무 많으면 부담)
- 선택지 제공 가능하면 제공 (예: "A인가요, B인가요?")
- 기술 용어보다 비즈니스 용어 사용
- 답변 없으면 다음 단계 진행 금지

## 행동 원칙

### DO
- **질문 단계 필수 수행**
- As-Is / To-Be 구조로 변경사항 정리
- 엣지 케이스 미리 도출
- 영향 받는 API, 코드, 데이터 모델 식별
- 작업 크기가 크면 더 작은 단위로 분해
- `docs/CONVENTIONS.md`의 API 응답 포맷 준수
- RESTful 원칙 준수
- 에러 케이스 빠짐없이 정의

### DON'T
- **질문 없이 바로 To-do 작성**
- 사용자 컨펌 없이 다음 단계 진행
- 기존 코드를 직접 수정하기
- 기존 API 스펙과 충돌하는 설계
- 직접 코드 작성 (설계만!)

> 설계 문서의 각 섹션별 예시는 [templates/design.md](templates/design.md) 참조
